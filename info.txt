# GitHub Workflow with Dev Branch

This document defines a clean, scalable GitHub workflow using a `dev` branch. It is designed for team collaboration, code quality, and controlled releases.

---

## 1. Branching Strategy

### Core Branches

- **main**
  - Production-ready code only
  - Always stable and deployable

- **dev**
  - Integration branch for active development
  - All features merge here first

---

### Supporting Branches

| Branch Type | Naming Convention | Purpose |
|------------|------------------|--------|
| Feature    | `feature/<name>` | New features |
| Bugfix     | `bugfix/<name>`  | Fixes for development |
| Hotfix     | `hotfix/<name>`  | Urgent production fixes |
| Release (optional) | `release/<version>` | Pre-production testing |

---

## 2. Development Workflow

### Step 1: Sync with `dev`

```bash
git checkout dev
git pull origin dev
````

### Step 2: Create a Feature Branch

```bash
git checkout -b feature/dashboard-ui
```

### Step 3: Develop and Commit

```bash
git add .
git commit -m "feat: implement dashboard UI cards"
```

### Step 4: Push to Remote

```bash
git push origin feature/dashboard-ui
```

---

## 3. Pull Request Workflow

* Create Pull Request:

  * `feature/*` → `dev`

### Requirements:

* Code review required
* CI checks must pass
* No direct merges

---

## 4. Release Workflow

### Option A: Using CLI

```bash
git checkout main
git pull origin main
git merge dev
git push origin main
```

### Option B: Using Pull Request

* Create PR: `dev` → `main`
* Review and merge

---

## 5. Hotfix Workflow

```bash
git checkout main
git pull origin main
git checkout -b hotfix/login-crash

# Apply fix
git add .
git commit -m "fix: resolve login crash on Android"

git push origin hotfix/login-crash
```

### Then:

* Create PR → `main`
* After merge → merge changes into `dev`

---

## 6. Commit Message Convention

| Type     | Example                           |
| -------- | --------------------------------- |
| feat     | `feat: add transaction filtering` |
| fix      | `fix: resolve crash on login`     |
| refactor | `refactor: clean up API service`  |
| style    | `style: adjust spacing`           |
| test     | `test: add unit tests`            |

---

## 7. Branch Protection Rules

### Protect `main`

* No direct pushes
* Require pull requests
* Require approvals
* Require CI checks

### Protect `dev` (Recommended)

* Require pull requests
* Require at least one review

---

## 8. Best Practices

### Keep Changes Small

* One feature per branch
* Easier to review and test

### Sync Regularly

```bash
git checkout dev
git pull origin dev
git checkout feature/your-branch
git merge dev
```

### Clean Up Branches

* Delete merged branches
* Enable auto-delete in GitHub

### Use Draft PRs

* For work-in-progress
* Gather early feedback

### Use CI/CD

* Run tests on all PRs
* Prevent broken merges

---

## 9. Workflow Overview

```
feature → dev → main
```

* Build in `feature/*`
* Integrate in `dev`
* Deploy from `main`

---

## 10. Common Mistakes

* Working directly on `main`
* Large PRs that are hard to review
* Not pulling latest `dev`
* Long-lived branches
* Skipping reviews

---

## 11. Example Branch Names

* `feature/date-range-filter`
* `feature/dashboard-redesign`
* `bugfix/login-error`
* `hotfix/payment-crash`

---

## 12. Recommended Setup

* Use `dev` as your working branch
* Require PR reviews
* Keep PRs focused (UI vs logic)
* Maintain clear commit messages

---

## 13. Future Enhancements

* Add `release/*` branches
* Implement automated deployments
* Consider trunk-based development for scaling
